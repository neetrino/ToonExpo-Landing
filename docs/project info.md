

# 📄 Описание проекта: Платформа строительного экспо

## 1. Общая концепция

Не очередной «сайт с карточками», а централизованная платформа для представления строительных компаний в рамках выставки (expo).

Система должна позволять:

* хранить данные о строительных компаниях (70–100+)
* автоматически генерировать для каждой компании отдельную публичную страницу (лендинг)
* предоставлять пользователям единый интерфейс для просмотра всех компаний

---

## 2. Основная логика системы

### 2.1. Компании (Core сущность)

В системе создается база строительных компаний.

Для каждой компании хранятся:

* название
* описание
* логотип
* фотографии (галерея)
* адрес (обязательно для карты)
* контактные данные
* дополнительные поля (по необходимости: услуги, проекты, ссылки и т.д.)

---

### 2.2. Генерация лендингов

Для каждой компании автоматически создается публичная страница (лендинг).

#### Принцип:

* один общий шаблон (единый дизайн и структура)
* меняется только контент (данные компании)

#### Лендинг включает:

* Hero (название + ключевое описание)
* Галерея (фото компании / проектов)
* Описание компании
* Контакты
* Адрес (с привязкой к карте)

👉 Важно:
лендинг не создается вручную — он динамически формируется на основе данных из базы.

---

### 2.3. Главная страница платформы

Главная страница — это навигационный хаб всей системы.

#### Содержит:

### 1. Карта

* отображает все компании
* каждая компания — маркер на карте
* при клике можно:

  * увидеть краткую информацию
  * перейти на лендинг компании

### 2. Список компаний

* список всех компаний (70–100)
* карточки с краткой информацией:

  * название
  * фото / логотип
  * короткое описание

### 3. Поиск / фильтрация

* поиск по названию
* (опционально) фильтры:

  * по категории
  * по локации

---

### 2.4. Переходы

* Клик по компании (в списке или на карте) → открывается лендинг этой компании

---

## 3. Админ-панель (минимальная)

Да, минимальная. Без попыток построить SAP за три дня.

### 3.1. Список компаний

* таблица / список всех компаний
* каждая строка = одна компания

### 3.2. CRUD функционал

Для компаний:

* ➕ Создание
* ✏️ Редактирование
* ❌ Удаление

### 3.3. Логика синхронизации

Любое действие в админке:

* автоматически влияет на публичную часть

| Действие          | Результат                |
| ----------------- | ------------------------ |
| Создание компании | появляется новый лендинг |
| Редактирование    | обновляется лендинг      |
| Удаление          | лендинг удаляется        |

---

## 4. Архитектурная логика (важно для разработчика)

Чтобы потом не переписывать всё «потому что не подумали»:

### 4.1. Тип системы

* CMS-подобная система
* data-driven rendering

### 4.2. Лендинги

* НЕ отдельные страницы вручную
* один шаблон + динамический роут:

```
/company/{slug}
```

### 4.3. Главная страница

* агрегирует данные из базы
* карта + список используют один источник данных

---

## 5. Требования к UX/UI

* единый дизайн для всех компаний
* аккуратный, современный, нейтральный стиль
* мобильная версия обязательна (expo → люди с телефонами)

---

## 6. Масштабируемость (чтобы потом не плакать)

Сразу учитывать:

* возможность увеличения до 200–300 компаний
* быструю загрузку списка и карты
* оптимизацию изображений

---

## 7. Итог

Система состоит из трех ключевых частей:

1. **База компаний**
2. **Автогенерация лендингов**
3. **Главная страница с картой и списком**
4. **Админка для управления**

---

Если коротко и без красивых слов:

Ты делаешь не 70 сайтов.
Ты делаешь **один сайт, который притворяется 70 сайтами**.

И если программист попытается сделать это иначе, лучше сразу остановить его, пока он не начал «каждой компании отдельный компонент руками».
